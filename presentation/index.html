상세 설명 — 프로젝트 개요 라이브러리

1️⃣ Stable Baselines3 (SB3)

강화학습 알고리즘(PPO, DQN, A2C 등)을 즉시 사용할 수 있는 PyTorch 기반 오픈소스 RL 라이브러리. 알고리즘 내부 구현 없이도 환경만 만들면 바로 학습을 돌릴 수 있어 RL 실험의 표준 도구로 쓰입니다. 학습 루프, 하이퍼파라미터 관리, 모델 저장/로드까지 통합 제공합니다.

프로젝트 적용: 탱커·딜러·힐러 3개 역할에 각각 PPO(policy="MlpPolicy", env=...)를 인스턴스화해 멀티 폴리시 구조 구현. 라운드 로빈 방식으로 역할당 300K 스텝씩 교대 학습하고, 이전 라운드 스냅샷 모델을 상대방으로 재활용하는 셀프 플레이 적용.

2️⃣ Gymnasium

OpenAI Gym의 후속 프로젝트로, 강화학습 환경의 표준 인터페이스를 정의하는 라이브러리. reset()으로 환경 초기화, step(action)으로 한 스텝 진행 후 (obs, reward, terminated, truncated, info)를 반환하는 구조를 표준화해 SB3 같은 RL 라이브러리가 어떤 환경과도 동일한 방식으로 상호작용할 수 있게 합니다. (terminated는 게임 종료, truncated는 최대 스텝 초과로 인한 강제 종료를 구분합니다.)

프로젝트 적용: BattleEnv와 NexusEnv가 모두 gymnasium.Env를 상속받아 구현됨. step() 내부에서 이동·공격·힐·자기장 피해·아이템 처리 + 보상 계산 수행. SB3의 PPO가 이 인터페이스를 통해 환경과 자동으로 상호작용.

3️⃣ TensorBoard

Google이 만든 머신러닝 학습 시각화 도구. 학습 중 보상(reward), 손실(loss), 정책 엔트로피 등의 지표를 실시간 그래프로 브라우저에서 확인할 수 있습니다. 학습이 잘 되고 있는지, 언제 과적합이 시작되는지 등을 빠르게 파악할 수 있습니다.

프로젝트 적용: SB3의 tensorboard_log="logs/" 파라미터로 자동 연동. 역할별(tank/dealer/healer) 학습 보상 추이와 정책 엔트로피 변화를 모니터링해 힐러 miss 페널티 강화, 딜러 쿨다운 조정 등 보상 리밸런싱 시점 결정에 활용.

상세 설명 — 관측 공간과 행동 공간

1️⃣ 관측 공간이란 무엇인가

강화학습에서 관측 공간(Observation Space)이란 에이전트가 매 스텝마다 환경으로부터 받는 정보의 집합입니다. 에이전트는 이 정보를 신경망에 입력해 어떤 행동을 할지 결정합니다.

본 프로젝트에서는 게임 상태 전체를 349개의 실수값(벡터)으로 수치화하여 에이전트에 전달합니다. 사람이 화면을 보고 상황을 판단하듯, 에이전트는 이 349개의 숫자로 현재 상황을 인식하고 행동을 결정하게 됩니다.

3️⃣ 4채널 9×9 시야맵 구조

에이전트를 중심으로 한 9×9 격자(81칸)를 목적에 따라 4개의 채널로 분리해 표현합니다. 각 칸의 값은 0.0(해당 없음)에서 1.0(해당) 사이의 실수입니다.

채널 0 — 지형: 벽이면 1.0, 자기장이면 0.5, 이동 가능한 빈칸이면 0.0
채널 1 — 아군: 해당 칸에 아군이 위치하면 1.0
채널 2 — 적군: 해당 칸에 적이 위치하면 1.0
채널 3 — 아이템: 아이템 종류에 따라 0.33 / 0.67 / 1.0으로 구분

81칸 × 4채널 = 324차원이 전체 관측의 약 93%를 차지합니다.

4️⃣ 행동 공간과 멀티 폴리시의 관계

신경망은 가능한 행동 각각에 대한 선택 확률을 출력하고, 그 중 가장 높은 확률의 행동을 실행합니다. 역할마다 유효한 행동이 다르기 때문에, 하나의 정책으로 통합할 경우 존재하지 않는 행동을 선택하는 invalid action 문제가 발생합니다. 이를 해결하기 위해 역할별 독립 정책(멀티 폴리시)을 채택했습니다.

탱커 (6가지): 상·하·좌·우 이동 / 근접 공격 / 대기
딜러 (7가지): 상·하·좌·우 이동 / 근접 공격 / 원거리 공격 / 대기
힐러 (7가지): 상·하·좌·우 이동 / 근접 공격 / 팀원 힐 / 대기

상세 설명 — DQN에서 PPO로 전환한 이유

1️⃣ 환경이 복잡해질수록 Q값 추정이 무너졌습니다

DQN은 가능한 모든 행동에 대해 "이 행동의 가치(Q값)"를 신경망으로 추정합니다. Phase 1(10×10 맵, 2명)처럼 단순한 환경에서는 문제없이 동작했지만, 맵이 15×15·20×20으로 커지고 자기장·아이템·팀 전투까지 더해지자 상태 공간이 폭발적으로 커져 Q값 추정 자체가 불안정해졌습니다.

프로젝트 전환 시점: Phase 1(DQN, 500K) → Phase 2에서 맵 확장 + 아이템·자기장 추가 직후 학습이 수렴하지 않고 보상이 요동치는 현상 발생 → PPO로 전환 후 안정화 확인

2️⃣ 과거 경험을 재활용할 수 없는 환경이었습니다

DQN의 핵심 기법인 Experience Replay는 과거 경험을 버퍼에 쌓아두고 무작위로 꺼내 재학습합니다. 이 방식은 환경이 변하지 않는다는 가정이 전제됩니다. 그러나 셀프 플레이에서는 상대 에이전트의 정책이 라운드마다 계속 바뀌기 때문에, 이전 라운드에 수집한 경험이 현재 상황을 제대로 반영하지 못하는 비정상성(Non-stationarity) 문제가 발생합니다.

PPO의 해결: PPO는 현재 정책으로 수집한 경험만 사용하는 on-policy 방식입니다. 라운드가 바뀌어 상대 정책이 달라져도, 그 시점에 수집한 최신 경험으로만 학습하므로 셀프 플레이 라운드 로빈 구조와 자연스럽게 맞아떨어집니다.

3️⃣ 역할별 독립 정책 3개를 교대 학습하는 구조가 필요했습니다

탱커·딜러·힐러 각각 독립된 정책을 300K 스텝씩 교대로 학습하는 라운드 로빈 구조에서, DQN은 역할마다 별도의 replay buffer를 유지해야 하고 상대 정책이 교체될 때마다 버퍼를 초기화하거나 오염 문제를 처리해야 합니다. 구현 복잡도가 높고 학습 신호가 섞이는 위험도 있었습니다.

PPO의 해결: 역할별로 PPO(...) 인스턴스를 3개 독립 생성하고, model.learn(300_000) 호출만으로 교대 학습이 완성됩니다. 구조가 단순하고 역할 간 학습 신호가 완전히 분리됩니다.

4️⃣ 병렬 환경(n_envs=8)이 학습 속도의 핵심이었습니다

역할당 3M 스텝이라는 대규모 학습을 현실적인 시간 안에 완료하려면 여러 환경을 동시에 실행해 경험을 병렬로 수집하는 것이 필수였습니다. DQN은 단일 환경 + replay buffer 구조가 기본이라 병렬화 시 버퍼 동기화 문제가 발생합니다.

PPO의 해결: SB3의 make_vec_env(n_envs=8)로 8개 환경을 병렬 실행, 한 번의 rollout에 8배 많은 경험을 수집해 학습 효율을 대폭 향상시켰습니다.

5️⃣ 보상 설계 실험 과정에서 안정성이 절실했습니다

힐러 miss 페널티 조정, 딜러 쿨다운 변경 등 보상 함수를 반복적으로 수정하며 실험하는 과정에서 DQN은 보상 스케일 변화에 민감해 하이퍼파라미터를 함께 재조정해야 하는 부담이 있었습니다. PPO의 clipping 기법은 정책이 한 번에 너무 많이 바뀌지 않도록 제한해, 보상 설계가 바뀌어도 학습이 급격히 붕괴되지 않고 안정적으로 수렴했습니다.

실제 사례: 힐러 허공 공격 문제 발견 후 miss 페널티를 -0.02 → -0.15로 7배 강화했을 때도 PPO clipping 덕분에 기존 학습 성과를 유지하며 점진적으로 행동이 교정되었습니다.

상세 설명 — 멀티 폴리시 · 셀프 플레이 · 라운드 로빈

1️⃣ 멀티 폴리시 (Multi-Policy)

하나의 신경망(정책)으로 모든 역할을 처리하는 대신, 역할마다 독립된 정책 네트워크를 따로 두는 방식입니다. 역할이 다르면 최적 행동 전략도 완전히 다르기 때문에, 하나의 정책에 서로 다른 학습 신호를 동시에 주면 정책이 어느 역할에도 특화되지 못하고 어정쩡해집니다.

프로젝트 적용: 탱커·딜러·힐러 각각 PPO(...) 인스턴스를 독립 생성. 행동 공간도 역할별로 분리(탱커 6가지·딜러 7가지·힐러 7가지)해 존재하지 않는 행동을 선택하는 invalid action 문제와 "데미지를 높여라(딜러 보상)" vs "팀원을 회복해라(힐러 보상)" 같은 모순된 학습 신호를 완전히 차단했습니다.

2️⃣ 셀프 플레이 (Self-Play)

학습 중인 에이전트가 자기 자신의 이전 버전과 대전하며 학습하는 방식입니다. 고정된 상대와 싸우면 그 상대에게만 특화되어 다른 전략에 취약해지지만, 셀프 플레이는 에이전트가 강해질수록 상대도 함께 강해지므로 지속적인 실력 향상이 가능합니다.

프로젝트 적용: 각 라운드 학습 시작 전, 직전 라운드에서 저장한 스냅샷 모델을 상대로 로드합니다. 예를 들어 탱커가 학습하는 동안 상대편 딜러·힐러는 이전 라운드 스냅샷으로 행동합니다. 학습이 진행될수록 상대 스냅샷도 강해지므로 에이전트가 점진적으로 더 어려운 상대와 대전하게 됩니다.

3️⃣ 라운드 로빈 학습 (Round-Robin Training)

여러 대상을 정해진 순서대로 돌아가며 공평하게 학습 기회를 주는 스케줄링 방식입니다. 한 역할이 학습하는 동안 나머지 역할은 고정된 스냅샷으로 행동하고, 순서가 돌아오면 다시 학습합니다. 이를 반복해 모든 역할이 균형 있게 성장합니다.

프로젝트 적용: Tank → Dealer → Healer 순서로 각 300K 스텝씩 학습, 이를 10라운드 반복.
라운드가 끝날 때마다 세 역할 모두 스냅샷을 저장하고, 다음 라운드에서 그 스냅샷을 상대로 활용합니다.
역할당 총 학습량: 300K × 10라운드 = 3M 스텝

상세 설명 — Phase별로 학습을 진행한 이유

1️⃣ 처음부터 복잡한 환경에서 학습하면 아무것도 배우지 못합니다

강화학습 에이전트는 보상을 받아야 학습합니다. 그런데 처음부터 20×20 맵에 6명·역할 시스템·자기장·아이템이 모두 존재하는 환경에 투입하면, 에이전트는 어떤 행동이 보상으로 이어지는지 연결고리를 찾지 못해 학습 신호 자체가 희박해집니다. 이를 해결하기 위해 단순한 환경에서 시작해 점차 복잡성을 높이는 커리큘럼 학습(Curriculum Learning) 방식을 채택했습니다.

2️⃣ Phase 1 — 기본 전투 행동부터 확인했습니다

10×10 소규모 맵에 에이전트 2명만 두고 DQN으로 학습시켰습니다. 목표는 고성능 에이전트가 아니라 "접근 → 공격"이라는 가장 기초적인 전투 패턴을 학습하는지 확인하는 것이었습니다. 환경이 단순할수록 보상이 자주 발생해 에이전트가 인과관계를 빠르게 파악할 수 있습니다.

확인한 것: 에이전트가 제자리에 머물지 않고 상대를 향해 이동하며 공격하는 기본 행동 학습 여부. 이 단계에서 DQN의 한계(불안정성)도 함께 확인해 PPO 전환의 근거를 마련했습니다.

3️⃣ Phase 2 — 한 번에 하나씩 요소를 추가했습니다

알고리즘을 DQN에서 PPO로 전환하면서, 아이템과 자기장을 동시에 추가했습니다. 새로운 요소를 한 번에 여러 개 추가하면 어떤 요소가 학습 불안정을 일으키는지 파악이 불가능합니다. Phase를 분리함으로써 각 변화가 학습에 미치는 영향을 독립적으로 관찰할 수 있었습니다.

확인한 것: PPO 전환 후 학습이 안정적으로 수렴하는지, 아이템·자기장이 추가된 환경에서도 전투 전략이 유지되는지 검증.

4️⃣ Phase 3 — 다인 환경에서의 생존 전략을 학습했습니다

20×20 맵으로 확장하고 에이전트를 5명(FFA)으로 늘렸습니다. 2명 환경과 달리 다인 전투에서는 언제 싸우고 언제 도망칠지라는 전략적 판단이 필요합니다. 이 단계에서 순위 기반 보상을 도입해 단순 전투 승리보다 생존을 우선하는 전략이 나타나기 시작했습니다.

확인한 것: 에이전트가 불리한 상황에서 교전을 피하고 자기장 밖 상대를 기다리는 도주·회피 전략이 자연발생적으로 출현. Phase 4 팀 전투 설계의 기반이 됐습니다.

5️⃣ Phase 4 — 앞선 학습의 기반 위에 역할 시스템을 도입했습니다

Phase 1~3에서 검증한 이동·공격·생존 패턴을 바탕으로, 팀 구성(3팀 × 2명)과 역할 시스템(탱커·딜러·힐러), 멀티 폴리시를 한꺼번에 도입했습니다. 이전 Phase들이 없었다면 역할별 보상 설계가 잘못됐는지, 환경 자체가 문제인지 원인을 분리해 디버깅하는 것이 불가능했을 것입니다.

핵심 교훈: "보상 하나를 바꾸면 행동 전체가 바뀐다"는 사실은 Phase별로 단계를 나눠 관찰했기 때문에 발견할 수 있었습니다. 단계 없이 최종 환경에서 바로 실험했다면 원인 파악 자체가 불가능했을 것입니다.

상세 설명 — 향후 발전 방향

1️⃣ RLlib — 왜 다음 단계에서 필요한가

RLlib은 Ray 프레임워크 기반의 분산 강화학습 라이브러리입니다. SB3이 단일 머신에서 소규모 실험에 최적화된 반면, RLlib은 수십 개의 CPU·GPU에 학습을 분산시켜 대규모 멀티 에이전트 환경을 효율적으로 다룰 수 있도록 설계되어 있습니다. 특히 에이전트 수가 늘어날수록 SB3의 구조적 한계가 드러납니다.

프로젝트에서 RLlib이 필요한 이유:
현재 SB3으로 역할별 PPO 3개를 직접 조율하는 라운드 로빈 구조는 수작업으로 구현된 것입니다. RLlib은 이런 멀티 에이전트 학습 스케줄링을 내장 지원하며, 에이전트 수·팀 수를 늘려도 코드 구조 변경 없이 확장이 가능합니다. 팀 수를 3팀에서 5팀·10팀으로 확장하거나, 역할을 추가할 때 SB3은 구조 자체를 재설계해야 합니다.

2️⃣ 커뮤니케이션 학습 — 에이전트끼리 신호를 주고받는다

현재 프로젝트에서 탱커·딜러·힐러는 각자의 관측 정보만으로 행동을 결정합니다. 팀원의 위치·HP 정도만 관측값으로 공유될 뿐, 서로 의도를 전달하는 수단이 없습니다. 커뮤니케이션 학습이란 에이전트가 행동을 결정하기 전에 다른 에이전트에게 학습 가능한 메시지 벡터를 전송하고 이를 반영해 행동을 결정하는 구조입니다.

기대 효과: 탱커가 "지금 공격 들어간다"는 신호를 보내면 힐러가 대기 상태에서 치료 준비로 전환하거나, 딜러가 "저 적은 내가 잡는다"는 신호로 역할 중복을 방지하는 등 현재 관측값만으로는 불가능한 협력 전략이 창발할 수 있습니다. QMIX, MAPPO 같은 알고리즘이 이 구조를 지원하며, RLlib에서 기본 제공됩니다.

3️⃣ 보상 자동 탐색 — 사람이 설계하지 않아도 되는 보상

현재 보상 함수는 "딜러 데미지 보상 2.2", "힐러 저체력 힐 7.0" 등 사람이 직접 수치를 정해 반복 실험으로 조정하는 방식입니다. 이 과정에서 의도치 않은 부작용이 자주 발생했습니다. 보상 자동 탐색은 보상 함수 자체를 학습의 대상으로 삼아 사람의 개입 없이 최적 보상을 찾아가는 방식입니다.

주요 접근 방식:
· RLHF (인간 피드백 강화학습): 사람이 에이전트 행동을 평가하면 그 피드백으로 보상 모델을 학습
· 역강화학습 (IRL): 전문가의 플레이를 관찰해 그 행동을 유도하는 보상 함수를 역으로 추론
· 자동 보상 형성 (Reward Shaping 자동화): 유전 알고리즘·베이즈 최적화로 보상 가중치를 자동 탐색

현재 프로젝트에서 힐러 miss 페널티를 -0.02에서 -0.15로 수동 조정했던 과정이 자동화될 수 있음을 의미합니다.
